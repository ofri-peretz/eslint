fix(docs): revert inappropriate security additions to non-security plugins

# Problem
Commit 1d68f96 incorrectly applied security-focused documentation standards
(CWE/OWASP/CVSS metadata and generic "Known False Negatives") to 160+
non-security rules across architectural/quality plugins:
- eslint-plugin-import-next (56 rules)
- eslint-plugin-architecture (14 rules)  
- eslint-plugin-quality (21 rules)
- eslint-plugin-react-features (45 rules)
- eslint-plugin-react-a11y (37 rules)

These plugins enforce code quality and architectural best practices, NOT
security vulnerabilities. Adding security metadata was inappropriate and
confusing.

# Solution
1. **Reverted documentation**: Restored all non-security plugin rule docs to
   their pre-1d68f96 state (removed CWE/OWASP/generic FN sections)

2. **Updated compliance audit**: Clarified `.agent/rules-compliance-audit.md`
   to explicitly state it applies to SECURITY plugins only. Added separate
   simplified structure for architectural/quality plugins.

3. **Preserved "Known False Negatives" requirement**: Per the
   `.agent/workflows/fn-documentation.md` standard, ALL rules (security and
   non-security) MUST include Known False Negatives documentation. The
   non-security structure now correctly includes this section.

4. **Fixed lint errors**: Removed duplicate export statements in
   - `apps/docs/scripts/sync-readme-rules.mjs`
   - `apps/docs/scripts/sync-rules-docs.mjs`

# Changes
- .agent/rules-compliance-audit.md:
  - Version bumped to 1.1

  - Added "Code Examples" to TOC
  - Created separate "For Non-Security Plugins" structure
  - Emphasized that Known False Negatives are required for ALL rules

- Rule documentation (160 files):
  - Removed inappropriate CWE/OWASP/CVSS/Severity metadata

- Scripts:
