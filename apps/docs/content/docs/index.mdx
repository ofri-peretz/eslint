## Why Traditional ESLint Falls Short in AI-Assisted Development

When you use AI assistants like **GitHub Copilot** or **Cursor**, traditional ESLint error messages create friction:

```
‚ùå Traditional ESLint Message (hard for AI to fix):
"Unexpected string concatenation" (no-template-curly-in-string)
```

AI assistants often:

- Don't understand the **security context**
- Can't determine **severity** or **priority**
- Generate **incomplete fixes** without knowing the vulnerability type
- Miss **compliance requirements** (SOC2, GDPR, PCI-DSS)

---

## How Interlace ESLint Changes the Game

Every rule produces a **structured 2-line format** optimized for AI understanding:

```
‚úÖ Interlace ESLint Message (AI-optimized):
üîí CWE-89 OWASP:A03-Injection CVSS:9.8 | SQL injection via string concatenation | CRITICAL
   Fix: Use parameterized query: client.query('SELECT * FROM users WHERE id = $1', [userId])
```

**Line 1 (Security Context):**

- **CWE ID** ‚Äî Weakness classification
- **OWASP Category** ‚Äî Top 10 mapping
- **CVSS Score** ‚Äî Severity (0-10)
- **Description** ‚Äî Clear problem statement
- **Severity Level** ‚Äî CRITICAL/HIGH/MEDIUM/LOW

**Line 2 (Actionable Fix):**

- **Concrete solution** with code example
- **No vague suggestions** ‚Äî precise remediation

## Quick Install

<Tabs items={['npm', 'pnpm', 'yarn']}>
  <Tab value="npm">
    ```bash npm install --save-dev eslint-plugin-secure-coding ```
  </Tab>
  <Tab value="pnpm">```bash pnpm add -D eslint-plugin-secure-coding ```</Tab>
  <Tab value="yarn">```bash yarn add -D eslint-plugin-secure-coding ```</Tab>
</Tabs>

## Configuration

```js title="eslint.config.js"
import secureCoding from 'eslint-plugin-secure-coding';

export default [secureCoding.configs.recommended];
```
