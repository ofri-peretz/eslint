## Description

Disallow the use of the `crypto-js` library. Modern Node.js and browser environments provide robust, performant, and secure built-in cryptographic APIs (Node.js `crypto` module and Web Crypto API) that should be preferred over third-party libraries like `crypto-js`.

## Rule Details

Detects imports and requires of `crypto-js` and flags them as insecure.

## Examples

### ‚ùå Incorrect

```javascript
import CryptoJS from 'crypto-js';
const hash = CryptoJS.SHA256('message');

const CryptoJS = require('crypto-js');
const ciphertext = CryptoJS.AES.encrypt(
  'my message',
  'secret key 123',
).toString();
```

### ‚úÖ Correct

```javascript
import crypto from 'node:crypto';
const hash = crypto.createHash('sha256').update('message').digest('hex');

// Using Web Crypto API
const msgUint8 = new TextEncoder().encode('message');
const hashBuffer = await crypto.subtle.digest('SHA-256', msgUint8);
```

## Options

This rule has no options.

## When Not To Use It

Only if you are in an environment that does not support the native `crypto` module or Web Crypto API (e.g., very old browsers or non-Node.js environments), which is rare for modern applications.

## Known False Negatives

The following patterns are **not detected** due to static analysis limitations:

### Aliased Modules

**Why**: If `crypto-js` is aliased in your build system (e.g., via Webpack resolve alias), it may be missed.

```typescript
// ‚ùå NOT DETECTED - if 'my-crypto' points to 'crypto-js'
import CryptoJS from 'my-crypto';
```

**Mitigation**: Audit your build configurations.

### Dynamic Require

**Why**: Dynamic requires are not always statically traceable.

```typescript
// ‚ùå NOT DETECTED
const name = 'crypto' + '-js';
const CryptoJS = require(name);
```

**Mitigation**: Use static imports/requires.
