# Accessibility

This document is the source of truth for accessibility (a11y) in `apps/docs`
and the components it consumes from `@interlace/ui`. It states what we test,
what passes, and which conventions every contributor must follow.

> **Bar**: zero `@axe-core` violations across **WCAG 2 A + AA + AAA, WCAG 2.1
> A + AA, WCAG 2.2 AA, axe-core best-practice, and ACT (Accessibility
> Conformance Testing) tags simultaneously**.

We deliberately exceed WCAG 2.1 AA — the most regulated baseline — by also
asserting AAA contrast and axe's own best-practice rules.

---

## How we test

| Tool | Where | What |
|---|---|---|
| `@axe-core/playwright` | [apps/docs/e2e/a11y.spec.ts](e2e/a11y.spec.ts) | Per-route axe scan. Strict tag stack. Fails the suite on any violation. |
| `apps/docs/scripts/a11y-summary.ts` | local + CI | One-shot triage. Produces `a11y-report.md` with a deduplicated, prioritized violation list. |
| `@axe-core/playwright` | reduced-motion test | Forces `prefers-reduced-motion: reduce` and re-scans home — catches motion-related regressions. |

### Run locally

```bash
# 1. start dev (or build + start production)
npm run -w docs dev

# 2. run the strict axe scan against running site
npm run -w docs test:a11y           # Playwright suite, fails on violations
npm run -w docs a11y:report         # markdown summary report
```

### CI lock

The Playwright test suite includes [a11y.spec.ts](e2e/a11y.spec.ts) — any
new violation, on any route, in any tag, blocks the merge.

## Color contrast — AAA

Every text/background combination on the site clears WCAG AAA (7:1 for
normal text, 4.5:1 for large/UI). Keep it that way.

### Brand tokens (defined in [src/app/global.css](src/app/global.css))

| Token | Light value | Dark value | Notes |
|---|---|---|---|
| `--color-fd-primary` | `hsl(263 80% 30%)` | `hsl(270 80% 80%)` | 7.7:1 / 8.5:1 |
| `--color-fd-foreground` | `hsl(0 0% 3.9%)` | `hsl(0 0% 95%)` | 19+:1 |
| `--color-fd-muted-foreground` | `hsl(0 0% 27%)` | `hsl(0 0% 75%)` | 7.4:1 / 7.5:1 |
| `--color-fd-accent-foreground` | `hsl(0 0% 9%)` | `hsl(0 0% 98%)` | 16+:1 |
| `--color-fd-ring` | `hsl(263 70% 45%)` | `hsl(270 75% 75%)` | UI 3:1+ |

If you change any brand token, the AAA scan must re-pass.

### Tinted-bg badges

For `bg-{color}-500/15` patterns (15%-tinted color over neutral bg), use
`text-{color}-900 dark:text-{color}-100` for AAA pass. Anything looser
(like `text-{color}-600`) only passes AA, not AAA.

### Code blocks

We use Shiki's **`github-light-default` / `github-dark-default`** themes
(higher-contrast variants of the older `github-light` / `github-dark`).
A small set of token colors that pass AA but fail AAA are CSS-overridden
in [global.css](src/app/global.css) under the **"SHIKI TOKEN-COLOR
BOOST"** block. If you add a new Shiki theme or new languages, re-run
the scan and add overrides for any new failing colors there.

## Reduced motion

Every animated component must degrade under `prefers-reduced-motion: reduce`.

- Pure CSS animations: **already disabled site-wide** by the global media
  query in [`@interlace/ui/styles/tokens.css`](../../packages/ui/styles/tokens.css)
  that sets `animation: none !important; transition: none !important` on
  every `animate-*` utility.
- JS-driven motion (motion/react, canvas): each component reads
  [`useReducedMotion`](../../packages/ui/src/lib/use-reduced-motion.ts) and
  early-returns or jumps to final state. Verified: NumberTicker,
  StarsBackground/ShootingStars/Meteors, BackgroundBeamsWithCollision,
  BackgroundLines.

A regression test forces `prefers-reduced-motion: reduce` on the home page
and re-scans for axe violations — see
[a11y.spec.ts](e2e/a11y.spec.ts).

| Page kind | Landmark structure |
|---|---|
| `/` (HomeLayout) | Single `<main id="nd-home-layout">` from fumadocs. Inner page wraps content in `<div>`, never `<main>`. |
| `/docs/...` (DocsLayout) | Outer `<main id="main-content" class="contents">` added in [src/app/docs/layout.tsx](src/app/docs/layout.tsx) — fumadocs DocsLayout doesn't ship one. |
| Code blocks | Each `<figure class="shiki">` and its inner `<div role="region">` get a unique `aria-label` injected by [`<CodeBlockLabeller>`](src/components/a11y/code-block-labeller.tsx) at the root layout level — satisfies `landmark-unique`. |

If you add a new layout, ensure exactly one top-level `<main>` per page —
no nesting, no duplicates.

## Component conventions enforced by axe

These are the violations we've burned in lessons from. Each maps to a
convention every contributor must follow.

### Always required

- **Every interactive element must have an accessible name.**
  - `<button>` with only an icon: `aria-label="..."`. Don't rely on
    `title` (axe at strict won't accept it).
  - `<input>` for free-text: an associated `<Label>`, not a placeholder.
  - `<a>` with only an icon child: `aria-label="..."`.
  - Custom triggers (Select, Combobox): explicit `aria-label`.
- **Every interactive element must be keyboard-reachable + operable.**
  Base UI handles this for behavioral primitives — verify in our
  compositions.
- **Every focus indicator must be visible.** Use the shared
  `focus-visible:` ring fragment from `@interlace/ui` primitives.
- **Decorative SVGs and icons get `aria-hidden="true"`**, never
  `role="img"` without an accessible name.
- **Image `alt`**: meaningful images get an `alt`; decorative images
  alongside text labels get `alt=""`. Never duplicate the adjacent text
  in the alt (`image-redundant-alt`).
- **One `<main>` per page.** Never nest, never duplicate.
- **Don't put `role="article"` on a clickable `<a>`**: anchors don't
  allow that role (`aria-allowed-role`).

### Required for our components specifically

- **`data-slot="<component-name>"`** on every primitive root — see
  [packages/ui/CONVENTIONS.md](../../packages/ui/CONVENTIONS.md).
- **`data-testid` on every component root and every meaningful interactive
  child** — see [Custom rule: data-testid for testability](#custom-rule-data-testid-for-testability) below.

## Custom rule: data-testid for testability

Every component that ships from `@interlace/ui` or lives in `apps/docs/src/components/`
must expose stable `data-testid` attributes for end-to-end testing.

### Why

- E2E tests (Playwright, Cypress) need stable selectors that survive class /
  styling refactors.
- Class-name selectors (`.hover\:bg-fd-muted`) are brittle — they break
  every time we tweak Tailwind utilities.
- Screen-reader-targeting selectors (`getByRole`, `getByLabel`) cover
  semantics but not visual structure assertions.
- `data-testid` gives us a stable identity that's invisible at runtime
  and untouchable by styling refactors.

### Where it's required

| Element | `data-testid` requirement |
|---|---|
| Component root | **Mandatory** — `data-testid="<component-name>"`. |
| Interactive children with state | **Mandatory** — toggles, inputs, triggers, the visible state-bearing parts. |
| Repeated items in a list | **Mandatory** — include an index or id (`page-{n}`, `article-card-{id}`). |
| Pure decorative wrappers | **Discouraged** — don't pollute. |

### Naming convention

- Lower-kebab-case.
- Hierarchical when nested: parent prefix + role:
  `pagination` → `pagination-prev`, `pagination-next`, `pagination-page-{n}`.
- Match the visible action verb where possible: `clear-search`, `toggle-theme`,
  `submit-form`.

### Examples

```tsx
// ✅ Good
<button data-testid="clear-search" aria-label="Clear search">
  <X aria-hidden />
</button>

<Pagination data-testid="article-pagination">
  <PaginationPrevious data-testid="article-pagination-prev" />
  <PaginationLink data-testid={`article-pagination-page-${n}`} />
</Pagination>

// ❌ Bad — relies on class name for selection in tests
<button className="hover:bg-fd-muted">Submit</button>
```

### Enforcement plan

- **Phase 1 (now)**: Convention documented here; reviewers enforce in PR.
- **Phase 2 (next)**: Custom ESLint rule
  `@interlace/eslint-plugin-conventions/require-data-testid` —
  flags any `<button>`, `<a>` (with click handler), `<input>`,
  `<select>`, `<textarea>`, or component root in `apps/docs/src/components/**`
  that lacks `data-testid`. Plumb into existing `eslint.config.mjs`.
- **Phase 3 (later)**: Lint rule for **stable identity** — `data-testid`
  values should be string literals, never dynamic (`{`item-${id}`}` is
  fine; `{Math.random()}` is not).

## When you change something visual

1. Run `npm run -w docs dev` (dev server)
2. Run `npm run -w docs a11y:report`
3. If the report shows ≥1 violation: fix before merging
4. If the change is to brand tokens / Shiki colors: also visually verify
   in Chrome devtools' contrast checker — axe is conservative; eyes
   sometimes catch what it misses

## Self-test philosophy — layered defense

**No single tool catches everything.** axe is the foundation, but it has
known blind spots. We use a layered model so a violation must escape
multiple independent checks to land in production.

### Why one layer isn't enough — the documented blind spots

axe-core (our automated baseline) does **not** detect:

- **Gradient backgrounds** — falls back on computed `background-color`
  (`transparent`), silently skips contrast checks. We hit this with the
  homepage's "Flat Config ✓" floating badge — passed axe, failed manual
  contrast math.
- **Pseudo-element content** — `::before` / `::after` decorations and their
  contrast.
- **Backdrop-filter / blur effects** — actual rendered color is opaque-blurred
  but axe sees the source.
- **Animation-state-only contrast** — text that only appears on hover or
  during a transition.
- **Focus-ring visibility against varied content** — axe checks the ring
  exists, not whether it's visible against every possible underlying color.
- **Reading order vs DOM order divergences** caused by CSS positioning.
- **Custom focus traps inside portals** — Base UI handles this, but
  hand-rolled overlays escape detection.
- **Cognitive load / readability** — sentence length, jargon, navigation
  predictability.

### The six layers

| Layer | What | When | Catches |
|---|---|---|---|
| **1. axe-core baseline** | `npm run -w docs test:a11y` | every CI run | structural ARIA, label, contrast (solid bg), heading order, landmark uniqueness |
| **2. Visual regression** | Playwright screenshot diffs (`test:e2e`) | every CI run | accidental color shifts, gradient + alpha contrast changes, focus-ring breakage |
| **3. Convention lints** | ESLint custom rules (`require-data-testid`, `no-low-contrast-tailwind`, `no-aria-hidden-on-focusable`) | edit-time + CI | low-contrast Tailwind combinations, missing test IDs, ARIA misuse before runtime |
| **4. Component isolation** | per-component story + axe-addon (Storybook later) | per-PR | regressions inside one component, before composition hides them |
| **5. Manual heuristics checklist** | the section below | every UI PR | gradient/blur blind spots, motion-only states, AT-narration sanity |
| **6. AT spot-checks** | VoiceOver / NVDA / TalkBack walkthrough | release branches | reading order, label clarity, real-world AT behavior |

### Mandatory at PR level

1. **axe scan must pass** (CI gate, locked).
2. **New colors require a contrast comment** at the source (e.g.,
   `// AAA: 7.7:1 on white`). PR review rejects un-measured color additions.
3. **New animated components require `useReducedMotion` integration** —
   verifiable by the reduced-motion test in [a11y.spec.ts](e2e/a11y.spec.ts).
4. **New gradients / backdrop-blur surfaces require a manual contrast
   measurement** at multiple stops, documented in the PR description. This
   covers axe's biggest blind spot.
5. **New components ship with a Storybook story** when Storybook is
   integrated (planned).

### The self-test checklist every contributor runs locally

Before opening a PR that changes UI, walk through this. A minute per item.

- [ ] **Tab through the page** — every interactive element is reachable in
      a sensible order; focus indicator is visible at every stop.
- [ ] **Press Escape** inside every overlay (dialog, sheet, popover,
      dropdown) — closes cleanly, focus returns to trigger.
- [ ] **Press / and Cmd+K** anywhere — the search/command palette opens.
- [ ] **DevTools → Rendering → Emulate `prefers-reduced-motion: reduce`** —
      no animations play, content is still legible.
- [ ] **DevTools → Rendering → Emulate `prefers-color-scheme: light` and
      `dark`** — every text/bg combination still passes contrast.
- [ ] **Resize to 320px width** — every interactive element is at least
      44×44px (WCAG 2.2 target size).
- [ ] **Resize text to 200%** (browser zoom or text-only zoom) — layout
      still works, no overflow clipping critical content.
- [ ] **Open DevTools → Accessibility → Color picker** — sample a few
      pixels from new gradients/blurs/decorations; verify ratio.
- [ ] **Disable JavaScript** — at minimum, the page renders meaningful
      static content; no critical info hidden behind required JS.
- [ ] **Read the page like a screen reader** — close your eyes and
      narrate what you'd hear: page title → landmarks → headings →
      paragraphs → interactive labels. Each link / button name must
      stand alone (no "click here"). Each image either has alt or is
      `aria-hidden`.

### Testing matrix (which layer covers which violation class)

| Violation class | Layer 1 axe | Layer 2 visual | Layer 3 lint | Layer 5 manual |
|---|---|---|---|---|
| Solid-bg contrast | ✅ | ✅ | partial | – |
| Gradient/blur contrast | ❌ | ✅ | ✅ (planned) | ✅ |
| Missing alt / aria-label | ✅ | – | ✅ (planned) | – |
| Heading order | ✅ | – | – | – |
| Landmark structure | ✅ | – | – | – |
| Keyboard reachability | ✅ | – | – | ✅ |
| Focus visibility | ✅ (presence) | ✅ (rendered) | – | ✅ |
| Reduced motion respect | ✅ (regression) | – | – | ✅ |
| Touch target ≥44×44 | partial | ✅ | – | ✅ |
| AT narration quality | – | – | – | ✅ |

A new violation class that escapes all five layers should trigger a
post-mortem and a process update — *the layers exist precisely to leave
an audit trail of what was supposed to catch what.*

### Anti-pattern: trusting axe alone

If you only run axe and ship green, you have **AA** — not AAA, and you
have *no* coverage for gradients, blurs, motion-only states, or AT
quality. Saying "axe passes" is the start of the conversation, not the
end. The badge above `Flat Config ✓` is the canonical example: axe
green, 3.37:1 actual. We caught it with eyes; the lint in Layer 3 will
catch the next one before it ships.
