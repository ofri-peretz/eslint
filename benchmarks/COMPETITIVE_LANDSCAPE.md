### Why these 18 (and not others)

- **Top-3 by relevance**: `eslint-plugin-security`, `sonarjs`, `microsoft-sdl` — the only plugins explicitly marketed as security tools with broad coverage.
- **Adjacent, high-install plugins included** (react, n, import, promise, regexp): teams running these see overlap with our coverage; we want to demonstrate complementarity.
- **Control group** (jsdoc, jest, jsx-a11y): plugins that shouldn't fire on our security fixtures. If they do, our fixtures are noisy.
- **Framework plugins** (vue, angular): show that framework lints don't substitute for security lints.

## ILB-AI — LLM-generated code security

**Source of truth:** [`benchmarks/ilb-ai/run-antigravity.js`](./benchmarks/ilb-ai/run-antigravity.js) — model registry
**Score:** vulnerability detection rate, Wilson 95% CI per (model × prompt category)
**Industry parallel:** HumanEval / SWE-Bench (task design); WMDP (security framing)

### The 10 LLM models tested

| # | Model | Provider | Access |
|---|---|---|---|
| 1 | Claude Opus 4.6 | Anthropic | API |
| 2 | Claude Sonnet 4.5 | Anthropic | API |
| 3 | Claude Haiku 4.5 | Anthropic | API |
| 4 | Gemini 3 Flash | Google | API |
| 5 | Gemini 3 Pro | Google | API |
| 6 | Gemini 2.5 Flash | Google | API |
| 7 | Gemini 3 Flash (CLI) | Google | `gemini` CLI |
| 8 | Gemini 3 Pro (CLI) | Google | `gemini` CLI |
| 9 | Gemini 2.5 Flash (CLI) | Google | `gemini` CLI |
| 10 | Gemini 2.5 Pro (CLI) | Google | `gemini` CLI |

### Methodology

Each model is asked 20 zero-context coding prompts (5 per category: database, auth, files, command exec, secrets). The generated code is linted with the **Interlace fleet** as the security telemetry. We measure:

- Per-model vulnerability rate (with 95% CI)
- χ² test: are models statistically distinguishable?
- Per-CWE breakdown
- Detection-vs-remediation success rate

**Latest result (Feb 2026):** Claude/Gemini families are statistically indistinguishable — insecure generation is a systemic property, not a model-specific flaw. Interlace detects 80–95% of injections across all models.

```bash
npm run ilb:ai -- --model=claude-sonnet-4.5
```

⚠️ **Costs real money** (LLM API tokens). One full run with 20 prompts × 10 models is roughly $5–$15 depending on subscription tier. A single-model run is $0.50–$1.50.

### Build the runner

When you're ready, the runner needs to:

1. Walk `corpus/CWE-*/{vulnerable,safe}/`
2. Lint each fixture with each competitor's `recommended` config

4. Compute per-CWE TP/FP/FN, F1, BAS
5. Emit `results/ilb-juliet/<date>.json` matching ILB-Arena's schema (so the scorecard can ingest it)

Estimated effort: 1 day (the FP/FN runner in ILB-Arena is mostly reusable).

## ILB-Cov — derived rule activation

Computed from `pluginRollup` in the latest ILB-Wild summary. No competitor matrix; it answers: *"of all rules across our 11 plugins, what fraction fired at least once on the corpus?"*

## Critical issue surfaced by the 2026-05-03 run

When we ran the full suite, **Interlace scored 0% recall on ILB-Arena and ILB-Arena-Quality** — but this is a **runner bug, not a detection failure**. Root cause:

- The bench's `interlace.config.js` does `import secureCoding from "eslint-plugin-secure-coding"`.
- npm workspaces resolves that to `packages/eslint-plugin-secure-coding/` (a symlink).
- That package's `package.json` declares `"main": "./src/index.js"` — but only `src/index.ts` exists. There is **no compiled JavaScript** in `packages/<plugin>/dist/` (Nx emits to repo-root `dist/packages/<plugin>/` instead).
- The bench runner (`spawnSync npx eslint --config interlace.config.js ...`) gets `ERR_MODULE_NOT_FOUND`, falls back to `[]` findings, and the runner records "0 TP" without surfacing the load failure.

**Confirmed via `/Users/ofri/repos/ofriperetz.dev/eslint/scripts/ilb-wild.mjs`** — that harness uses `tsx` to load TS sources directly and lints **2,106 / 1,942 / 923** findings respectively on three.js / webpack / vercel-ai. Same plugins, same recommended configs, real findings. So the rules work; only the ILB-Arena runner is broken for our plugins.

**Fix options** (in order of preference):

1. **Rewrite ILB-Arena's runner** to use the ESLint Node API + `tsx`-loaded ESM imports (~1 day; mirror what `ilb-wild.mjs` does).
2. Build the plugins to `packages/<plugin>/dist/` and update each `package.json` `main` to `./dist/src/index.js`.
3. Publish each plugin to npm and pin the bench's deps to specific versions (defeats "test the local code").

Option 1 is the cleanest — it makes the bench mirror the harness already proven to work in `ilb-wild`.

Until this is fixed, ILB-Arena's published Interlace numbers should carry the disclaimer: **"Interlace 0% is a runner artifact pending fix, not a detection result."**
