# ILB Submission Protocol v1 — Public Benchmark Leaderboard

> **Roadmap.** [LEADERSHIP_ROADMAP.md](../LEADERSHIP_ROADMAP.md) §3.1.
> **Status (2026-05-09).** Spec + publisher + storage shipped. Hosted UI deployment is the only remaining external action (DNS + Vercel/Cloudflare account work).

The Interlace Lint Bench (ILB) is the first public **submission framework** for JavaScript / TypeScript static-analysis tools. Tools submit a SARIF run on the canonical corpus; the framework computes scores; the leaderboard ranks them. Same model as MLPerf / SWE-Bench / OWASP Benchmark — but for JS/TS lint.

## 1. Why submit

- **Credibility.** Public, reproducible numbers anyone can verify. Beats marketing claims.
- **Apples-to-apples.** Same corpus, same scoring formula, same toolchain block. No cherry-picking.
- **Free positioning.** Your tool gets a row on the public leaderboard alongside Interlace, CodeQL, Semgrep, Snyk Code, sonarjs.

## 2. What you submit

A single JSON file matching [`benchmarks/lib/result-schema.json`](../lib/result-schema.json) (the Interlace bench-result envelope), plus a separate SARIF v2.1.0 file with the per-finding evidence.

Required envelope fields:

```jsonc
{
  "bench":            "ILB-Public-Submission",          // fixed for all submissions
  "benchVersion":     "1.0",                            // matches the corpus version
  "tool":             { "name": "your-tool", "version": "1.2.3", "homepage": "https://..." },
  "submitter":        { "name": "Your Name", "email": "you@example.com", "affiliation": "Your Co" },
  "timestamp":        "2026-05-09T15:00:00.000Z",
  "methodologyCommit": "<SHA of the ILB corpus commit you ran against>",
  "toolchain":        { /* node, eslint, typescript, tsCompiler, platform — full schema in result-schema.json */ },
  "corpus":           { "version": "1.0", "kind": "ILB-Arena | ILB-Juliet | ILB-Wild" },
  "sarifPath":        "./your-tool-2026-05-09.sarif",   // sibling SARIF file, relative to envelope
  "effectiveness":    { /* computed by the publisher from your SARIF; do not pre-fill */ },
  "latency":          { "meanLatencyMs": 12345, "msPerFile": 0.7 }
}
```

The `effectiveness` block (F1, precision, recall, ranks) is **computed by the publisher** against ground-truth labels in the corpus. You don't supply it — that prevents self-reporting bias.

## 3. How to submit

Two channels, both work:

1. **PR to `interlace-bench/submissions`** — open a PR adding your envelope + SARIF under `submissions/<your-tool>/<YYYY-MM-DD>/`. CI runs the publisher; the leaderboard auto-rebuilds.
2. **API POST** (Phase 3.1+ once hosted): `POST /api/submit` to the leaderboard endpoint with multipart `envelope.json` + `result.sarif`.

Either way: the publisher recomputes scores against the ground truth, validates the toolchain block is complete, and rejects submissions that fail the [vocabulary contract](../README.md#three-measurement-aspects-vocabulary-contract).

## 4. The corpus your tool will be scored against

Three corpora, run independently:

| Corpus | Scope | Ground truth | Score |
| :--- | :--- | :--- | :--- |
| **ILB-Arena** | 40 vulnerable + 38 safe synthetic fixtures | per-fixture `@expected` annotation | F1 + rank |
| **ILB-Juliet** | CWE-mapped corpus (CWE-022, -078, -079, -089, -798, -918) | per-fixture `@cwe` + `@expected` | F1 per CWE + OWASP BAS |
| **ILB-Wild** | 22 popular OSS repos pinned to commits (1.7M LoC) | finding density, no labels | findings/kLoC + per-rule attribution |

Corpora are versioned. Bumping any pinned commit / fixture / scoring formula bumps the corpus version (principle #9). Submissions are scored against the version they declared they ran against.

## 5. What the leaderboard reports

For each submission:

- **F1 weighted by CVSS** (headline, see [stats.mjs#weightedF1](../lib/stats.ts))
- F1 plain + bootstrap CI
- Precision · Recall · per-CWE breakdown
- Latency: `meanLatencyMs` · `msPerFile` · `peakRssMb`
- Toolchain pinning (Node / ESLint / TS compiler / platform)
- **Disagreement matrix vs. peers** — where you differ from everyone else is the most interesting signal

Publication artifact: [`benchmark-results/leaderboard.md`](../../benchmark-results/leaderboard.md), regenerated by `npm run ilb:leaderboard:publish`.

## 6. Anti-cheat

- **Pre-registration**: every submission must declare `methodologyCommit`. We verify it exists in the corpus repo and matches the corpus version it claims to score against.
- **Reproducibility**: your submission must include a runnable script that produces the SARIF from a fresh checkout. We re-run a random 10% of submissions monthly to catch drift.
- **Toolchain pinning**: `toolchain.tsCompiler` and `toolchain.eslint` are validated; submissions with `unknown` are tagged but not ranked alongside pinned-toolchain submissions.
- **Ground-truth lockdown**: corpus labels live in a separate repo (`interlace-bench/corpus`, governed independently per Phase 4 item 4.5). Submitters never see the ground-truth before scoring.

## 7. Submission lifecycle

```
1. Tool author runs Interlace bench corpus → emits SARIF
2. Author drafts envelope.json (toolchain + tool metadata + sarifPath)
3. Author opens PR to interlace-bench/submissions OR POSTs to /api/submit
4. Publisher computes scores against ground truth
5. Validator checks toolchain block + methodologyCommit + vocabulary contract
6. Leaderboard rebuilds. Tool gets a row.
7. Optional: re-run for monthly verification.
```
