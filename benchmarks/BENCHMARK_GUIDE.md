# Benchmark Guide — what each bench measures, where the evidence lives

> **For:** anyone reading the [`scorecard`](../../benchmark-results/scorecard.md), [`RESULTS.md`](./RESULTS.md), or a leaderboard table.
> **Goal:** explain in plain English what each bench actually tests, what TP/FP/FN/F1 mean in concrete terms, and where the per-rule receipts live so any number can be drilled into.

---

## How the suite is organized

We have **seven benches**. Each one answers exactly one question, so the right number is always the right size.

| # | Bench | Question it answers |
|---|---|---|
| 1 | **ILB-Arena** | "If we put us next to every other ESLint security plugin, who wins?" |
| 2 | **ILB-Arena-Quality** | Same, but for code-quality plugins. |
| 3 | **ILB-Juliet** | "On the standard CWE-mapped synthetic corpus (the academic test), do we detect the patterns?" |
| 4 | **ILB-Wild** | "What do our plugins find on real popular OSS projects?" |
| 5 | **ILB-Edge** | "Where do we false-positive on legitimate code that *looks* dangerous?" |
| 6 | **ILB-Cov** | "What % of our rules ever fire on real code?" |
| 7 | **ILB-AI** | "Do our rules catch security flaws in code generated by LLMs?" |

You should **read each one in isolation** — they don't combine into one super-score. Different benches answer different questions.

## Glossary — what TP/FP/FN/F1/BAS actually mean

Every accuracy bench scores findings against **ground truth** (someone hand-labeled which fixtures *should* fire and which *shouldn't*). The four buckets:

| Term | Plain English | Example |
|---|---|---|
| **TP** (True Positive) | Plugin flagged a real vulnerability ✅ | Plugin flagged `eval(userInput)` — correct, this is RCE |
| **FN** (False Negative) | Plugin missed a real vulnerability ❌ | Plugin didn't flag `'SELECT * FROM users WHERE id=' + userId` — missed SQL injection |
| **FP** (False Positive) | Plugin flagged safe code as vulnerable ❌ | Plugin flagged `eval('1 + 1')` (literal arg, no user input) — wrong |
| **TN** (True Negative) | Plugin correctly ignored safe code ✅ | Plugin didn't flag `db.query('SELECT $1', [userId])` — correct, parameterized |

Then:

```text
Precision = TP / (TP + FP)   "When it cries wolf, was it a wolf?"
Recall    = TP / (TP + FN)   "Did it catch every wolf?"
F1        = harmonic mean of the two — single comparable number, 0–100%
BAS       = TPR − FPR        OWASP Benchmark Accuracy Score (Youden's J)
```

**Higher is better** for all four.

## Bench 1 — ILB-Arena

### What it tests

A **head-to-head** between 18 ESLint security plugins on the same 78 fixtures (40 vulnerable + 38 safe).

### Where the evidence lives

| Layer | Path |
|---|---|
| Fixture (vulnerable code) | [`benchmarks/ilb-arena/fixtures/vulnerable/vulnerable.js`](./benchmarks/ilb-arena/fixtures/vulnerable/vulnerable.js) — 40 functions, each labeled `vuln_<category>_<pattern>` (e.g. `vuln_sql_string_concat`) |
| Fixture (safe code) | [`benchmarks/ilb-arena/fixtures/safe/safe-patterns.js`](./benchmarks/ilb-arena/fixtures/safe/safe-patterns.js) — 38 functions, each labeled `safe_<category>_<pattern>` |
| Ground truth | The function names themselves (`vuln_*` should fire, `safe_*` should not) |
| Per-plugin config | [`benchmarks/ilb-arena/configs/<plugin>.config.js`](./benchmarks/ilb-arena/configs/) — 18 files, one per competitor |
| Runner | [`benchmarks/ilb-arena/run.js`](./benchmarks/ilb-arena/run.js) |
| Latest result | [`results/ilb-arena/2026-05-03.json`](./results/ilb-arena/2026-05-03.json) — full `vulnerableAnalysis` + `safeAnalysis` objects per plugin |

### How to read the leaderboard table

```text
| Plugin                          | TP  | FP | FN | Precision | Recall | F1     |
| Interlace ESLint Ecosystem      | 40  | 1  | 0  | 97.6%     | 100.0% | 98.8%  |
```

**This row says:**
- Out of 40 vulnerable patterns we *should* catch, we caught **40 (100% recall — we missed nothing)**.
- Out of 38 safe patterns we *shouldn't* flag, we wrongly flagged **1** (`FP=1`).
- F1 score: **98.8%**, the single comparable number.

### What the same metrics show for the closest competitor

```text
| eslint-plugin-sonarjs (SonarSou | 14  | 5  | 26 | 73.7%     | 35.0%  | 47.5%  |
```

- Sonarjs caught only 14 of 40 vulnerable patterns (**recall 35%** — missed 26).
- Of its 14 detections, 5 were on safe code (FP rate ~13%).
- F1 47.5%, half of ours.

### How to dig into specific findings

**Drill into a specific FN** ("which 26 did sonarjs miss?"):

```bash
# In benchmarks/results/ilb-arena/2026-05-03.json
.plugins["sonarjs"].vulnerableAnalysis.functions[?].findings == []
# Functions with empty findings = false negatives
```

**Drill into a specific TP/FP** ("what rule caught this fixture?"):

The result file's `vulnerableAnalysis` and `safeAnalysis` objects map each fixture function to the rule(s) that fired on it. So you can answer "which rule of ours caught `vuln_sql_string_concat`?" → it'll say `pg/no-unsafe-query`.

### Per-plugin coverage on this fixture set

The 40 vulnerable fixtures are organized by category. Where each Interlace plugin shows up:

| Category | Vulnerable fixtures (count) | Plugins responsible |
|---|---|---|
| SQL Injection (CWE-89) | 5 | `pg`, `secure-coding` |
| Command Injection (CWE-78) | 4 | `secure-coding`, `node-security` |
| Path Traversal (CWE-22) | 4 | `node-security`, `secure-coding` |
| Hardcoded Credentials (CWE-798) | 4 | `secure-coding`, `node-security` |
| JWT (CWE-321/325/347) | 3 | `jwt` |
| XSS (CWE-79) | 4 | `browser-security` |
| Logging Injection | 2 | `secure-coding` |
| Prototype Pollution (CWE-1321) | 5 | `secure-coding` |
| Insecure Randomness (CWE-330) | 3 | `crypto`, `secure-coding` |

So when ILB-Arena reports "Interlace TP=40", it means **every plugin in the fleet contributed**: pg caught its 5, jwt caught its 3, crypto caught its 5, etc. The result file's `vulnerableAnalysis` shows which rule of which plugin fired on each fixture — that's the audit trail.

## Bench 3 — ILB-Juliet

### What it tests

The same TP/FP scoring, but on a **CWE-organized corpus**. Each CWE has its own folder with vulnerable/ and safe/ fixtures. This is what NIST SARD/Juliet and OWASP Benchmark Project do for Java/C++; we're doing it for JavaScript.

### Where the evidence lives

```text
benchmarks/corpus/
├── CWE-022/   Path Traversal
│   ├── vulnerable/  (e.g. fs.readFile(req.params.path) — should fire)
│   ├── safe/        (e.g. fs.readFile(path.join('/safe', basename(input))) — should NOT fire)
│   └── manifest.json   { name, owasp, severity, expectedPlugins }
├── CWE-078/   OS Command Injection
├── CWE-079/   Cross-Site Scripting (XSS)
├── CWE-089/   SQL Injection
```

## Bench 4 — ILB-Wild

### What it tests

Our plugins running against **real, popular GitHub OSS projects** (pinned to specific commits). This isn't a competition — there's no ground truth on real OSS — it's an **exposure** measure: how much do our rules surface in the wild?

### Where the evidence lives

```text
benchmark-results/<date>/
├── summary.json + summary.md           cross-repo aggregate
└── per-repo/<repo-name>/
    ├── run.json                        timing + RSS + LoC + finding counts
    ├── per-rule.json                   ruleId → { hits, avgTimeMs, samples[] }
    ├── findings-sample.json            top 50 raw findings (file, line, msg)
    └── report.md                       rendered per-repo report
```

The **per-rule.json** is the audit trail. For example, on `vercel/ai`:

```text
{
  "secure-coding/detect-object-injection": {
    "hits": 421,
    "avgTimeMs": 12.3,
    "samples": [
      { "file": "packages/ai/core/util/parse-partial-json.ts", "line": 87, ... }
    ]
  }
}
```

## Bench 5 — ILB-Edge

The 5 🔬 repos above. **Findings on these projects default to "FP candidate pending triage"** — by design, they use risky-looking patterns legitimately:

- three.js → `eval()`, `new Function()` for WebGL shaders
- webpack → `eval()` for HMR + `Function` for runtime modules
- react-dom → `dangerouslySetInnerHTML` is the API
- lodash/fp → dynamic property access is the library
- babel → `Function` constructor + AST property access in the parser

**4,106 candidate FPs** to triage. Every one we confirm as a real FP becomes a rule refinement. This is the queue that improves the noise floor.

The same per-repo artifacts apply (run.json, per-rule.json, samples). [`benchmark-results/2026-05-03/per-repo/three.js/per-rule.json`](../../benchmark-results/2026-05-03/per-repo/three.js/per-rule.json) is the receipt.

## What this all means

- **On synthetic fixtures (Arena + Juliet)**: we win, decisively. F1 98.8% on Arena, 76.5% on Juliet — vs sonarjs at 47.5% / 40%.
- **On real OSS (Wild)**: we surface real signals (vercel/ai 923) and stay quiet on clean code (sst, react-dom, babel — 0 findings).
- **On adversarial-real (Edge)**: we have 4,106 FP candidates to triage — the noise floor we need to drive down.
- **On performance**: 13.5 ms/file median is well under the 15 ms SLO.

What's pending: full triage of FP candidates, expanding the CWE corpus from 6 → 40+, and running the AI bench with current-month models.
