**Reproducible benchmarks for the Interlace ESLint ecosystem.** This is the **single source of truth** for how we measure, value, and act on lint-rule accuracy.

## 1. Philosophy — how we value FP / FN / TP / TN

Every ESLint finding is one of four things:

| | **Code IS vulnerable / problematic** | **Code is safe / clean** |
| :--- | :--- | :--- |
| **Rule fires** | **TP** — true positive (signal) | **FP** — false positive (noise) |
| **Rule silent** | **FN** — false negative (miss) | **TN** — true negative (correct quiet) |

We collapse those four counts into three published rates:

- **Precision** = TP / (TP + FP) — *of the things we flag, what fraction are real?*
- **Recall** = TP / (TP + FN) — *of the things we should flag, what fraction did we catch?*
- **F1** = harmonic mean of precision and recall — *one number that punishes either being bad.*

### The ten principles

1. **Recall first, precision second.** A missed CWE is worse than a noisy rule. We never regress recall to chase FPs. Today: **100% recall on Arena, CWE-Corpus** — guard it.
2. **Two axes of truth, never one.** Synthetic corpora (Arena, CWE-Corpus) prove labels; real OSS (Wild, Edge) proves production fidelity. Neither alone is sufficient — both must be green.
3. **Adversarial framing is explicit.** FP-edge repos (three.js, webpack, lodash) are *expected* to over-fire. We mark them `fpEdge: true` in the registry and judge them on FP-rate ceilings, not zero-FP.
4. **Per-rule attribution is mandatory.** No "we have N FPs" without knowing which rule each came from. Every aggregate must drill to a rule + fixture + line. Our tooling enforces this.
5. **Multi-rater agreement beats single-author labels.** Cohen's κ vs sonarjs / microsoft-sdl is a trust signal. The over-fit detector (only-Interlace TPs) is its dual — both run in `ilb:coverage`.
6. **Trust through reproducibility.** Every claim has a JSON receipt and a `jq` query that recomputes it. Markdown is a window onto JSON, not a separate source.
7. **Drift is a bug.** Stale benches are observability failures, not "scheduling issues." The CI gate fails the PR; the audits/ folder timestamps each refresh.
8. **FPs are weighted by bench severity.** An Arena FP is publication-grade harm; an Edge FP is expected drift. The regression gate weights them: **Arena ×10, CWE-Corpus ×5, Quality ×3, Wild ×1, Edge ×0.1**.
9. **Consistency over time builds credibility.** TP/FP/FN/TN counts must be **comparable across runs**. That requires (a) a frozen corpus per bench version (no silent commit drift — enforced by `npm run ilb:corpus-integrity`), (b) deterministic rule input (no env-dependent behaviour), (c) append-only history in `benchmark-results/history.ndjson` so any number can be plotted across time. Every result publishes the bench *version* so corpus changes invalidate prior numbers explicitly, not silently.
10. **Severity is a contract, not a label.** Every rule's severity (`error` / `warn`) defines what action a maintainer expects when it fires. The classification is rule-driven, not vibe-driven — see *"Severity classification policy"* below.

### Severity classification policy

A rule's severity must match its real-world consequence and false-positive risk. Use this table when authoring a new rule or auditing an existing one:

| Severity | When to use | Required FP profile | Examples |
| :--- | :--- | :--- | :--- |
| **`error`** | Code is **broken or exploitable as written.** A reviewer who sees this rule fire is expected to **block the PR.** | **Precision ≥ 95%** on Wild + Arena. False positive here trains devs to ignore CI failures — worse than not having the rule. | Hardcoded credential in production source · `eval(userInput)` · SQL string concatenation · prototype pollution sink · CWE Top-25 with confirmed taint flow |
| **`warn`** | Code is **likely problematic in some contexts** and a reviewer should look but may legitimately keep it. Default for security rules where context determines impact. | **Precision ≥ 70%** on Wild. Some FPs are acceptable because the dev is the adjudicator. | `detect-object-injection` (depends on whether `key` is user-controlled) · `no-insecure-comparison` (depends on whether crypto context) · most `secure-coding/*` rules |
| **`info`** *(reserved)* | Stylistic / informational; no security or reliability impact. | n/a | — *(we don't currently use this tier; reserved for future use)* |
| **`off`** | Rule is `experimental`, `deprecated`, or specific to a workflow most users don't have. Available but not in `recommended`. | n/a | New rules during the lifecycle gate (Phase 1 day-7 in the audit roadmap) |

**Promotion rule:** A rule cannot ship as `error` until it has ≥ 4 fixtures across Arena + CWE-Corpus AND ≥ 90 days of Wild data showing the precision threshold met. New rules ship `warn`. Promotion to `error` is a deliberate decision recorded in the rule's CHANGELOG, not a default. Demotion is mandatory if Wild precision drops below threshold for two consecutive nightly runs.

**The cost of miscalibration:** every time CI fails on a low-precision `error`-level rule, devs learn that "CI failures don't always mean broken." That habit, once formed, is the most expensive thing in any lint setup. Severity is the only lever we have to keep this contract honest.

### The three measurement aspects (vocabulary contract)

Every Interlace bench reports — at minimum — these three aspects so cross-bench comparisons stay coherent. **Use this exact vocabulary in every result JSON, every scorecard cell, every CI gate, every audit doc.** No synonyms, no per-bench renames; if a bench doesn't measure one of the three, say so explicitly rather than substituting another label.

| Aspect | Standard field name(s) | What it answers | SLO framing |
| :--- | :--- | :--- | :--- |
| **Cost (tokens)** | `tokensO200k` (headline) · `tokensCl100k` (sanity) · `meanTokensO200k` (aggregate) | Will an LLM consumer choke on this? Is feeding it back into a fix loop cheap enough to do per-PR? | Per-bench ceiling (e.g. `interlace-compact ≤ eslint-stylish` for ILB-Formatter; `≤ V1` for ILB-LLM-Tokens). |
| **Effectiveness (context)** | `effectiveness` (umbrella) — concrete: `f1` / `precision` / `recall` (rule benches) · `signalScore` 0–4 (formatter) · `passRate` (ILB-LLM-Fix) · `vulnDetectionRate` (ILB-AI) | Does the output preserve the four FP/FN-attribution axes (`ruleId`, `severity`, `count`, `fixable`)? Did the rule classify TP/FP correctly? Did the LLM ship a working fix? | F1 ≥ 95 % (Arena) · F1 ≥ 80 % (CWE-Corpus) · structured signalScore = 4.0/4 (Formatter) · pass rate ≥ 80 % (LLM-Fix). |
| **Latency (speed)** | `latencyMsP50` · `latencyMsP95` · `meanLatencyMs` · per-rule `msPerFile` (perf benches) · `peakRssMb` | Can it run in a pre-commit hook? Does CI fit the time budget? Does the formatter still feel instant on a 2k-file run? | ≤ 15 ms/file (ILB-Perf) · ≤ 50 ms/run at `large` scale, ≤ 250 ms at `extreme` (ILB-Formatter). |

The triad is checked **constantly**, not occasionally: every result JSON in `results/<bench>/<date>.json` carries all three aspects (or an explicit `n/a` with reason), and the scorecard surfaces the per-aspect cell so a regression on any one of them fails the gate.

**Why all three matter:** a formatter that's cheap on tokens but loses signal is useless; a rule that's accurate but slow blocks adoption; a rule that's fast and accurate but produces noisy output trains devs to ignore CI. Optimising one aspect at the expense of another is the classic anti-pattern this triad exists to prevent.

### The provenance contract

A bench number is only as trustworthy as the surface it ran against. Every result JSON in this repo therefore carries an explicit **`provenance`** block answering the two reproducibility questions a sceptical reader always asks:

1. **Which model produced (or scored) the output?**
2. **Which tools were involved, at which versions, in what role?**

Without these, a "75 % pass rate" or a "−90 % token cost" is unfalsifiable: a re-run on a different model or tokenizer can land anywhere and there's no way to tell which of model, code, or fixture moved.

#### Required schema

```jsonc
"provenance": {
  "model": {
    "kind": "llm" | "tokenizer-proxy" | "none",   // one of these three — no others
    "name": "claude-haiku-4-5" | "o200k_base" | ...,
    "role": "what the model is doing in this bench (e.g. \"emits fix candidates\" / \"cost-axis surrogate\")"
  },
  "tools": {
    "count": <integer — total distinct tools listed>,
    "items": [
      { "name": "<package or binary>", "version": "<exact version>", "role": "<what it does in this run>" }
      // … one entry per tool — see "what counts as a tool" below
    ]
  },
  "subjectsUnderTest": [...],   // bench-specific list of what's being judged (formats, plugins, models)
  "comparators": [...]          // bench-specific list of what we're measured against (baseline + competitors)
}
```

#### Rules

- **`model.kind` is required.** A bench that doesn't call an LLM uses `"tokenizer-proxy"` (the tokenizer stands in for the model's cost surface) or `"none"`. There is no "didn't think about it" tier.
- **Every tool that materially affects the result is listed.** Test runner / formatter / tokenizer / linter / external CLI — yes. Node's stdlib — no. The litmus test: *"if I swap this for a different version, could the headline number move?"* If yes, list it.
- **`tools.count` must match `tools.items.length`** (the runner asserts this). Drift between count and list is a stale-result smell.
- **Versions are resolved at runtime**, not hand-rolled. We read `<pkg>/package.json#version` so the JSON reflects what actually loaded.
- **Roles are descriptive, not generic.** "tokenizer for cost axis (`o200k_base` headline + `cl100k_base` sanity)" — yes. "tokenizer" — no.
- **Multi-model benches list every model.** ILB-LLM-Fix runs `haiku × sonnet × opus` → three `model` entries (or one block with a `models[]` list, scored independently per model).

#### Why this is separate from `versions`

`versions` already records the package versions that loaded. `provenance` adds the **roles + counts + model surface** so a reader doesn't have to reverse-engineer the bench from a flat list of versions. They're different objects on purpose.

### The fixture metadata contract

Every fixture under [`corpus/`](./corpus/) carries:

```js
// @author        ofri-peretz       // who wrote it
// @reviewedBy    benchmark-validator // independent second pair of eyes (must differ from @author)
// @lastReviewed  2026-05-03          // YYYY-MM-DD
// @cwe           CWE-089             // ground truth label
// @expected      vulnerable | safe   // what every tool *should* say
```

`npm run ilb:validate-fixtures:strict` (wired into `npm run quality`) fails CI if any of these are missing or if `@author === @reviewedBy`.

## 3. Industry parallels — why we use these methodologies

Where standards exist, we mirror them. Where none exist for JavaScript, we say "we define this."

| Methodology | Adopted in | What we mirror |
| :--- | :--- | :--- |
| **NIST SARD / Juliet Test Suite** | layout inspiration only — **not** adopted as a scored corpus | `corpus/CWE-NNN/{vulnerable,safe}/` directory layout and per-CWE manifests follow Juliet conventions. We do **not** score against the actual NIST Juliet test suite; our `ILB-CWE-Corpus` is a self-authored 22-vuln + 22-safe synthetic set (orders of magnitude smaller than Juliet). Integrating a real NIST Juliet adaptation is tracked as Tier 0.4 in the strategic plan. |
| **OWASP Benchmark v1.2** | scoring methodology mirrored in ILB-CWE-Corpus + ILB-Arena | We compute BAS (TPR − FPR) and truth-table-per-CWE in the same shape as OWASP Benchmark v1.2 publishes them. **We do not run against the OWASP Benchmark corpus itself** — Benchmark v1.2 is a Java Web App fixture set; the JS adaptation does not exist publicly. Methodology-mirror, not corpus-mirror. |
| **MITRE CWE Top 25** | corpus prioritization | We expand the CWE corpus prioritizing Top-25 entries |
| **CVSS 3.1** | per-rule metadata | Every Interlace rule carries a CVSS score; results report findings by CVSS |
| **Wilson Score Interval** | all accuracy benches | 95% CI on F1 / precision / recall — required for n < 30 |
| **Chi-squared (χ²) test** | ILB-AI | Test whether different LLMs produce statistically distinguishable vulnerability rates |
| **MLPerf Inference** | ILB-Perf | Cold/warm/incremental scenarios, median-of-N runs, peak RSS, versioned submissions |
| **HumanEval / SWE-Bench** | ILB-AI | Zero-context prompt isolation, structured prompt categories, per-model statistical comparison |
| **Adversarial GLUE / CheckList** | ILB-Edge | Curated *legitimate*-but-tricky inputs; findings default to FP candidates pending triage |

ILB-Wild and ILB-Cov have no JavaScript predecessor — we publish the corpus and version the registry, and pitch the methodology as a candidate community standard.

## 4. Per-bench detail

### ILB-Arena — security head-to-head, 18 plugins

- **Corpus:** 40 vulnerable + 38 safe fixtures, frozen
- **Competitors:** 18 — `eslint-plugin-security`, `sonarjs`, `microsoft-sdl`, `unicorn`, `n`, `import`, `regexp`, `no-secrets`, `no-unsanitized`, `security-node`, `react`, `jsx-a11y`, `promise`, `jest`, `vue`, `angular`, `jsdoc`, **Interlace**
- **Score:** F1 with Wilson 95% CI, ranked
- **ESLint matrix:** runs against every supported major (v8, v9, v10) — see [`suites/ilb-arena/README.md`](suites/ilb-arena/README.md) and the [version support policy](../docs/ESLINT_VERSION_SUPPORT.md). A competitor that crashes under one major but not another counts as a comparison data point, not an outage.
- **Run:** `npm run ilb:arena` (~2 min) · single plugin: `npm run ilb:arena -- --plugin=interlace`
- **Output:** `results/ilb-arena/{eslint8,eslint9,eslint10}-<date>.json`

### ILB-Arena-Quality — quality head-to-head, 8 plugins

- **Corpus:** clean + problematic fixtures across the quality concerns (reliability, maintainability, modernization, conventions, modularity, operability)
- **Competitors:** 8 — `sonarjs`, `unicorn`, `import`, `promise`, `jsdoc`, `n`, `regexp`, **Interlace Quality Fleet**
- **Run:** `npm run ilb:arena:quality` (~1 min)
- **Output:** `results/ilb-arena-quality/<date>.json`

### ILB-CWE-Corpus — self-authored synthetic CWE accuracy

- **Corpus:** [`corpus/CWE-022/`](./corpus/CWE-022/) (path traversal), `CWE-078` (cmd injection), `CWE-079` (XSS), `CWE-089` (SQL-i), `CWE-798` (hardcoded creds), `CWE-918` (SSRF) — 2 vulnerable + 2 safe per CWE minimum
- **Score:** F1 per CWE + aggregate, plus OWASP BAS = TPR − FPR (Youden's J)
- **Competitors:** 6 — `eslint-plugin-security`, `security-node`, `sonarjs`, `microsoft-sdl`, `no-unsanitized`, **Interlace**
- **Run:** `npm run ilb:cwe-corpus` (~30s) · single CWE: `npm run ilb:cwe-corpus -- --cwe=CWE-089`
- **Output:** `results/ilb-cwe-corpus/<date>.json`

### ILB-Wild — real-OSS exposure

- **Corpus:** 22 popular OSS repos pinned to specific commits — twentyhq, supabase, three.js, webpack, vercel/ai, serverless, sst, payload, langchain-js, nestjs, aws-lambda-powertools, medusa, shadcn-ui, babel, cal.com, strapi, react, appwrite, lodash, nestjs-typeorm, next.js, serverless-api-gateway-caching. Plugin → repo mapping in [`BASELINE_MATRIX.md`](./BASELINE_MATRIX.md).
- **Score:** findings / kLoC + plugin coverage % + per-repo density
- **Run:** `npm run ilb:wild` (~30 min cold) · single repo: `npm run ilb:wild -- --repo three.js` · matrix: `npm run ilb:wild -- --list`
- **Output:** `../benchmark-results/<date>/per-repo/<repo>/`

### ILB-Edge — FP resilience on adversarial-real code

- **Corpus:** subset of Wild marked `fpEdge: true` — three.js, webpack, lodash, react, babel. These intentionally use patterns that *look* dangerous (eval for HMR, dynamic property access for bundling) but are correct in context.
- **Score:** FP rate (post-triage) — ceiling, not zero
- **Run:** auto-derived from `npm run ilb:wild`
- **Output:** included in the Wild per-repo reports

### ILB-Cov — rule activation

- **Score:** % of total rules that fire at least once across the Wild corpus
- **Why:** surfaces dead rules and plugins-without-target-corpus
- **Run:** auto-derived from `npm run ilb:wild`

### ILB-Perf — throughput

- **Scenarios:** cold cache, warm cache, incremental
- **Score:** ms/file (median of 3 runs) + peak RSS
- **Run:** auto-derived from `npm run ilb:wild` · import-specific: `npm run ilb:perf:import`

### ILB-AI — vuln detection on LLM code

- **Corpus:** 20 prompts × N models (Claude family, Gemini family) — zero-context, no security hints
- **Score:** vulnerability rate with Wilson 95% CI; χ² for cross-model comparison
- **Cadence:** quarterly (API budget) · last run: 2026-02-09
- **Run:** `npm run ilb:ai` (requires `ANTHROPIC_API_KEY` / `GOOGLE_API_KEY`)

### ILB-LLM-Tokens / ILB-LLM-Fix — per-finding messages for LLM consumers

- **Tokens:** token cost of our per-finding diagnostic output across messaging variants (V1 vs V2 modes) — answers "are individual findings cheap to feed back to an LLM?"
- **Fix:** does an LLM correctly fix code given our diagnostic? — answers "is the diagnostic actionable?"
- **Run:** `npm run ilb:llm:tokens` (no API) · `npm run ilb:llm:fix` (calls Claude, ~$5)
- **Scope note:** these benchmarks measure the **per-finding messaging layer** (`@interlace/eslint-devkit`). For the **whole-run formatter** that ESLint calls with `-f`, see ILB-Formatter (#10).

### ILB-Formatter — whole-run output cost + FP/FN signal preservation

- **Corpus:** 5 shapes (`mono-rule-storm`, `diverse-rules`, `mixed-severity`, `all-fixable`, `security-heavy`) × 5 scales (`tiny`/`small`/`medium`/`large`/`extreme`, 1 → 2,000 files), generated deterministically from a seeded RNG
- **Formats:** `interlace-{human,compact,json}` vs ESLint built-ins (`stylish` baseline, `json`, `json-with-metadata`, `html`)
- **Score:** mean token-cost delta vs `eslint-stylish` (`o200k_base`) + signal-preservation score (0..4: `ruleId`, `severity`, `count`, `fixable`)
- **SLO:** `interlace-compact` cost ≤ 0 % vs stylish · structured formats (`interlace-json`, `eslint-json`, `eslint-json-with-metadata`) must score 4.0 on every fixture (CI gate)
- **Why it matters for FP/FN:** the FP/FN philosophy below requires *per-rule attribution*. A formatter that drops `ruleId` / `severity` / fixable / count silently breaks every downstream LLM-fix prompt, dashboard, and regression gate. This bench treats the four axes as a hard contract on structured formats.
- **Run:** `npm run ilb:formatter` (no API) — output: `results/ilb-formatter/<date>.json`

## 6. How to read a verdict — `jq` drill recipes

Every claim above is recomputable from a JSON file. Five common questions:

```bash
# 1. "What's the current Arena leaderboard?"
jq '.summary.leaderboard' benchmarks/results/ilb-arena/$(ls benchmarks/results/ilb-arena/ | tail -1)

# 2. "Which CWE-Corpus fixtures FP?" (per CWE + fixture + rule)
jq '.plugins.interlace.perCwe | to_entries[] | {cwe: .key, fps: [.value.fixtures[] | select(.expectedVulnerable==false and .findings>0) | {file, ruleHits}]}' \
  benchmarks/results/ilb-cwe-corpus/$(ls benchmarks/results/ilb-cwe-corpus/ | tail -1)

# 3. "Quality FPs broken down by rule"
jq '.plugins."interlace-quality".cleanAnalysis.byRule' \
  benchmarks/results/ilb-arena-quality/$(ls benchmarks/results/ilb-arena-quality/ | tail -1)

# 4. "Quality fixtures we still miss (FNs)"
jq '.plugins."interlace-quality".problematicAnalysis.byFunction | to_entries | map(select(.value | length == 0)) | map(.key)' \
  benchmarks/results/ilb-arena-quality/$(ls benchmarks/results/ilb-arena-quality/ | tail -1)

# 5. "Top noisy rules across all real OSS"
jq -s 'map(to_entries) | flatten | group_by(.key) | map({rule: .[0].key, hits: map(.value.hits) | add}) | sort_by(.hits) | reverse | .[:20]' \
  benchmark-results/$(ls benchmark-results/ | grep '^2' | tail -1)/per-repo/*/per-rule.json

# 6. "Formatter cost vs eslint-stylish + which signal axes drop on which format"
jq '.headlineScore, .signalContract, (.summary | map({format, meanTokensO200k, meanSignalScore, meanGroupCollapse}))' \
  benchmarks/results/ilb-formatter/latest.json

# 7. "Which (shape, scale) cells lose a signal axis?"
jq '.measurements | map(select(.signal.score < 4)) | map({format, shape, scale, signal})' \
  benchmarks/results/ilb-formatter/latest.json
```

## 7. Statistical rigor

The minimum-viable disclosure for any number we publish:

- **Wilson Score interval** (not normal approximation) for proportions — required for n < 30
- **χ² test** for categorical comparisons (e.g. cross-model in ILB-AI)
- **Confidence intervals** on every published rate (F1, precision, recall, vuln rate)
- **No "blended scores."** F1 is F1; ms/file is ms/file. Cross-bench summaries live in `scorecard.md`, not in any single bench
- **Per-finding metadata** in every result JSON — `ruleId`, `cwe`, `cvss`, `owasp`, `line`, `sourceLine`

Wilson formula:

```
center = (p + z²/2n) / (1 + z²/n)
margin = z × √(p(1-p)/n + z²/4n²) / (1 + z²/n)
CI = [center - margin, center + margin]    where z = 1.96 for 95% confidence
```

If a result claims "Model A is 5 points better than Model B" and the CIs overlap, it's **statistically indistinguishable** — we say so explicitly.

## 8. Adding things — fixtures, benches, rules

### Adding a fixture

1. Pick the right CWE folder under [`corpus/`](./corpus/) (or create one — see "adding a CWE")
2. Add the file under `vulnerable/` or `safe/`
3. Stamp the metadata header (`@author`, `@reviewedBy`, `@lastReviewed`, `@cwe`, `@expected`)
4. Update the CWE manifest (`corpus/CWE-NNN/manifest.json`) — list the file under `fixtures` with its expected verdict
5. Run `npm run ilb:validate-fixtures:strict` — must pass before commit

### Adding a CWE corpus

1. `mkdir -p benchmarks/corpus/CWE-NNN/{vulnerable,safe}`
2. Write `manifest.json` declaring `expectedPlugins`, `coverageGap` (if any), and the fixture list
3. Add **at least 2 vulnerable + 2 safe** fixtures (Wilson CI gets too wide below this)
4. Re-run `npm run ilb:cwe-corpus` and check the per-CWE F1 is what you expect

### Adding a bench

The naming + structural rules:

1. **Single dimension only.** If it measures two things, it's two benches.
2. **Single-number top-line score.** Comparable across versions.
3. **`ILB-<short>` + version.** `ILB-Watch v1.0`, etc.
4. **Industry parallel cited or "we define this."** No false claims.
5. **Frozen corpus per version.** Bump the version when the corpus changes.

Then create `suites/ilb-<name>/`, add a runner emitting to `results/ilb-<name>/<date>.json`, and wire `npm run ilb:<name>` into `package.json`. Add a row to the scorecard generator (`scripts/ilb-scorecard.ts`).

### Adding a rule

New rules ship `experimental` (in plugin, not in any preset). They graduate to `recommended` automatically when:

- ≥ 80% precision on Wild, AND
- 0 FPs on Arena, AND
- the precision number has been stable for ≥ 14 days

Promotion to `strict` is manual.

## 10. Reproducibility

Every published number is reproducible from a fresh clone:

```bash
git clone https://github.com/ofri-peretz/eslint
cd eslint
npm install
npm run ilb:scorecard      # reads latest results — no fresh run
npm run ilb:wild           # produces a fresh ILB-Wild against pinned commits
npm run ilb:arena          # head-to-head security comparison
npm run ilb:regression     # gate against benchmark-results/baseline.json
```

If a number in this repo doesn't match what you reproduce, open an issue with the corpus version and your environment (`node --version`, `process.platform`).

### Where target OSS repos get cloned (ILB-Wild)

| Priority | Path | When |
| :--- | :--- | :--- |
| 1 | `$ILB_OOS_DIR` (env var) | CI / custom layouts |
| 2 | `~/repos/ofriperetz.dev/oos/` | local development default |
| 3 | `<repo>/.bench-repos/` | fallback if home-dir layout is non-standard |

```bash
ILB_OOS_DIR=/tmp/oss-cache npm run ilb:wild       # one-off
echo "export ILB_OOS_DIR=$HOME/oss" >> ~/.zshrc   # persistent
```

The bench is portable across Linux / macOS / Windows + CI/local — no hardcoded user paths.
