## The four type-aware rules (full detail)

### 🟠 `eslint-plugin-import-next/named` — graceful

- **What it does**: Verifies named imports (`import { foo } from 'bar'`) resolve to a real exported binding.
- **Why types**: Symbol resolution to the imported module's symbol table.
- **Without types**: `if (!hasParserServices(context)) return {}`. Rule is silent on JS-only projects and inside oxlint's JS-plugin tier.
- **Source**: [`packages/eslint-plugin-import-next/src/rules/named.ts`](../packages/eslint-plugin-import-next/src/rules/named.ts)

### 🟠 `eslint-plugin-import-next/namespace` — graceful

- **What it does**: Validates `ns.member` access against the imported module's exports.
- **Why types**: Walks the symbol table to find members of the namespace import.
- **Without types**: Silent (no-op).
- **Source**: [`packages/eslint-plugin-import-next/src/rules/namespace.ts`](../packages/eslint-plugin-import-next/src/rules/namespace.ts)

### 🟠 `eslint-plugin-import-next/default` — graceful

- **What it does**: Verifies the imported module has a default export when used via `import x from 'pkg'`.
- **Why types**: Symbol resolution + alias chasing for re-exports.
- **Without types**: Silent (no-op).
- **Source**: [`packages/eslint-plugin-import-next/src/rules/default.ts`](../packages/eslint-plugin-import-next/src/rules/default.ts)

### 🟡 `eslint-plugin-secure-coding/detect-object-injection` — refining

- **What it does**: Flags potentially unsafe bracket-property access (`obj[userInput]`).
- **Why types**: Drops FPs when the index is provably a union of safe string literals or a numeric type.
- **Without types**: Pure-AST detection still runs; precision drops slightly without the type narrowing.
- **Source**: [`packages/eslint-plugin-secure-coding/src/rules/detect-object-injection/index.ts`](../packages/eslint-plugin-secure-coding/src/rules/detect-object-injection/index.ts)

## Detection methodology

The classification is produced by static analysis of every rule source under `packages/eslint-plugin-*/src/rules/`. A rule is flagged as type-aware (any mode) if its source contains any of:

```
getTypeChecker(            getTypeAtLocation(         getParserServices(
hasParserServices(         parserServices.program     program.getType
import.*from 'typescript'  ts.Program                 ts.TypeChecker
@typescript-eslint/type-utils                         meta.docs.requiresTypeChecking
```

A rule is downgraded from `🔴 hard` to `🟠 graceful` if it also contains a `hasParserServices(context)` guard with an early return. It's downgraded further to `🟡 refining` if the type-info path is gated by `if (hasTypeInfo)` while pure-AST detection runs unconditionally.

To reproduce:

```bash
# regenerate the scan
bash scripts/scan-type-awareness.sh > .agent/type-awareness-scan.tsv

# regenerate the audit summaries from the scan
node scripts/build-type-awareness-audit.mjs   # writes .agent/type-awareness-audit.md
```

## Maintenance

- **Updated automatically by** the scan script when it runs (target: nightly via the bench workflow).
- **Manually verified after**: any PR that adds, removes, or modifies a rule's `create()` function.
- **CI gate**: `npm run audit:type-awareness` (planned) — fails if a rule's classification changed without an entry in the changelog or the philosophy doc's §3 ("when type-awareness earns its keep").
- **Drift policy**: the percentage of type-unaware rules must stay ≥ 95%. A drop below triggers a roadmap re-evaluation.
