<summary>Per-plugin notes + uncovered examples</summary>

### `eslint-plugin-node-security` (70%)

- **Target surface:** Node.js core modules — fs, child_process, crypto, vm, dns, http(s) (node@24 stable APIs)
- **Notes:** Coverage focuses on the high-frequency-misuse exports (`fs.readFile`/`writeFile`, `child_process.exec`, `crypto.create*`, `vm.runInThisContext`). The 30% gap is mostly low-traffic APIs where misuse is exceedingly rare.
- **Uncovered examples:**
  - node:vm.SourceTextModule — niche, not in 99% of code paths
  - node:dns.lookup variants — covered by adjacent rules only
  - node:http.Agent — TLS config patterns are covered, agent reuse less so
  - node:tls.createSecureContext — partially covered via require-secure-context

### `eslint-plugin-pg` (93%)

- **Target surface:** node-postgres — pg.Client / pg.Pool / pg.Query callable surface (pg@8.x)
- **Notes:** Near-complete coverage of the query-construction surface. The single uncovered API is type-parser registration, which is not a SQL-injection sink.
- **Uncovered examples:**
  - pg.types.setTypeParser — type-coercion sink, not a SQL injection surface; correctly out of scope

### `eslint-plugin-mongodb-security` (73%)

- **Target surface:** mongodb driver — Collection / Db / cursor query methods (mongodb@6.x)
- **Notes:** Coverage spans the high-frequency query construction surface (find, update, delete, $where). bulkWrite + aggregate need expanded coverage; both already detected.
- **Uncovered examples:**
  - Collection.bulkWrite — covered for find/update operators, not for indexes
  - Db.command — raw command pass-through, low-frequency
  - Collection.aggregate — stage pipeline checks, only the most common stages

### `eslint-plugin-jwt` (100%)

- **Target surface:** jsonwebtoken — sign / verify / decode + jose / fast-jwt analogues (jsonwebtoken@9.x, jose@5.x)
- **Notes:** Full coverage of the public sign/verify/decode surface across the three dominant JWT libraries.

### `eslint-plugin-express-security` (64%)

- **Target surface:** Express — router methods, middleware patterns, common response sinks (express@4.x + express@5.x)
- **Notes:** Coverage spans the high-traffic response sinks and CORS / cookie / CSRF middleware patterns. Template-engine and view-render rules are a known follow-up area.
- **Uncovered examples:**
  - app.engine — view-engine misconfig (rare)
  - res.render — template-injection sinks (partial)
  - app.set('trust proxy') — IP-handling correctness
  - express.Router merge-params — option misuse

### `eslint-plugin-lambda-security` (72%)

- **Target surface:** AWS Lambda — handler signatures, env-var access, callback / context patterns, AWS SDK client misuse (@aws-sdk/* v3 + Lambda Node runtime)
- **Notes:** Coverage spans the entry-point patterns (handler shape, env-var leak, secret-in-env), IAM role patterns, and the AWS SDK v3 client construction surface. Response streaming is an emerging surface to expand.
- **Uncovered examples:**
  - Lambda extensions API — niche
  - Lambda response streaming — newer pattern, partial coverage
  - context.identity (Cognito JWT) — covered indirectly via jwt plugin

### `eslint-plugin-nestjs-security` (63%)

- **Target surface:** NestJS — Controllers, Providers, Guards, Decorators, Pipes (nestjs@10.x + nestjs@11.x)
- **Notes:** 63% is at the floor. NestJS coverage is the lightest of the security plugins (6 rules). High-value expansion targets: ExceptionFilter patterns and Microservices transport-layer security.
- **Uncovered examples:**
  - Custom Pipes — narrowly covered (validation-pipe shape only)
  - ExceptionFilters — minimal coverage
  - Interceptors — coverage of error-leak patterns only
  - Microservices transports — not yet covered (Kafka / NATS / Redis)

### `eslint-plugin-vercel-ai-security` (75%)

- **Target surface:** Vercel AI SDK — generateText / streamText / generateObject / tools (ai@3.x + @ai-sdk/* providers)
- **Notes:** 75% on a moving target (the AI SDK ships quickly). 19 rules on 12 APIs reflects that some APIs have multiple distinct misuse patterns we cover separately.
- **Uncovered examples:**
  - useChat / useCompletion hooks — client-side patterns covered shallowly
  - RSC streamUI — newer; partial
  - Multi-step tool-call orchestration — covered in pieces, not as a unified flow

### `eslint-plugin-browser-security` (71%)

- **Target surface:** Browser web platform — DOM sinks, postMessage, storage, fetch, WebSocket, ServiceWorker, IndexedDB (WHATWG / WHATWG-fetch / W3C as of 2026)
- **Notes:** 71% across a large surface (58 APIs). Coverage prioritizes the high-frequency XSS / CSRF / data-leak sinks. Newer APIs (WebTransport, Compute Pressure) are tracked but uncovered.
- **Uncovered examples:**
  - Push API — niche
  - Web Share API — niche
  - WebRTC signaling — covered for postMessage layer, not RTCPeerConnection sinks
  - WebTransport — newer, not yet covered
  - Permissions API — partial

### `eslint-plugin-secure-coding` (95%)

- **Target surface:** Generic JS — eval / Function / regex / serialization / credentials (ECMAScript 2024)
- **Notes:** Near-complete coverage of the generic OWASP-Top-10 surface. 27 rules on 19 APIs reflects multiple distinct misuses per API.
- **Uncovered examples:**
  - Worker / SharedWorker — host-side sinks, partial
