# Ecosystem Landscape — per-plugin map

> Good competition is great. This document maps each Interlace plugin against
> the neighbors already doing strong work in the same space, so we can be
> honest about where we add depth, where someone else already covers a need
> well, and where the community has an open opportunity.
>
> Framing rules (per `memory/feedback_landscape_not_threats.md`):
> we describe **the landscape and our path through it**, not a battle
> board. Every entry leads with what neighbors do well, then where Interlace
> adds depth. Engines (ESLint, Oxlint, Biome, TSC native) are runtimes we
> ship to, not entries on this list — see `INTEROP_PHILOSOPHY.md`.
>
> Rule counts are from `.agent/oxlint-jsplugins-manifest.json` (authoritative,
> regenerated by `scripts/generate-oxlint-shims.mjs`). Weekly npm download
> numbers for neighbor plugins are approximate (2026-02 refresh) — use
> `npm run stats:eslint-versions` to refresh. Last updated: 2026-05-13.

## Reading guide

The lint ecosystem has three layers. This document covers layer 3.

1. **Engines** — ESLint, Oxlint, Biome, TSC native plugin host. Substrates
   we ship to via the parity contract. Not peers in this doc.
2. **Stock rule corpora inside engines** — Oxlint's 790 rules across
   15 namespaces (audited 2026-05-14 against `oos/oxc@3b46a8d`),
   Biome's ~400 built-ins. Overlap with our plugins is summarized
   per-entry below; the rule-namespace audit lives in
   [OXLINT_STOCK_OVERLAP.md](./OXLINT_STOCK_OVERLAP.md) and
   [BIOME_STOCK_OVERLAP.md](./BIOME_STOCK_OVERLAP.md).
3. **Plugins** (this document) — fellow rule libraries in the ESLint
   plugin ecosystem.

SAST analyzers (CodeQL, Semgrep, Snyk Code) sit on a different layer
(deep, periodic analysis) and are covered by the public
[/docs/getting-started/concepts/compare](../apps/docs/content/docs/getting-started/concepts/compare.mdx) page.

## Security plugins (our 10)

The security vertical is where Interlace specializes most deeply. For every
generic security plugin in the ecosystem, we add one or more dedicated
verticals (database, runtime, framework, SDK).

### `eslint-plugin-secure-coding`

**Our scope:** general secure coding practices and OWASP-aligned rules for
JavaScript/TypeScript (no eval, no hardcoded credentials, ReDoS detection,
PII handling, etc.).

**Neighbors in this space:**
- `eslint-plugin-security` (~1.6M weekly) — the long-standing generic
  security plugin. ~13 rules, broad surface.
- `eslint-plugin-no-secrets` (~250k weekly) — focused on credential leaks.
- `eslint-plugin-xss` (abandoned).
- `eslint-plugin-sonarjs` (~1.9M) — overlaps in a small security subset.
- Oxlint stock has a small `security/` namespace.

**Where we add depth:** per-rule CWE/CVSS metadata, MCP integration for
agentic remediation, autofixers where the fix is deterministic. Our
secure-coding pairs with our 9 domain-security plugins so users move from
generic to domain as they outgrow the basics. Engine portability via the
parity contract (runs under Oxlint today).

---

### `eslint-plugin-browser-security`

**Our scope:** browser/web-platform attack surfaces — XSS via innerHTML,
unescaped URL params, postMessage wildcard origins, websocket scheme
validation, CSP-adjacent rules.

**Neighbors in this space:**
- `eslint-plugin-no-unsanitized` (Mozilla, ~150k weekly) — narrow but
  well-maintained, covers `innerHTML`/`outerHTML`/`document.write` sinks.
- A small subset of `eslint-plugin-jsx-a11y` touches URL safety.

**Where we add depth:** broader sink coverage beyond innerHTML, structured
postMessage/websocket rules, CWE mapping. This is a moderately uncrowded
neighborhood — community opportunity to lead.

---

### `eslint-plugin-node-security`

**Our scope:** security for Node.js core modules — `fs`, `child_process`,
`crypto`, `vm`, `dns`, plus the crypto rules consolidated here in 2026-05.

**Neighbors in this space:**
- `eslint-plugin-n` (~5.4M weekly) — broad Node best-practices plugin,
  some security-adjacent rules (no-deprecated-api, shebang). General
  Node hygiene, not security-focused.
- `eslint-plugin-security-node` (low maintenance) — small overlap.
- `eslint-plugin-crypto` (deprecated, do not recommend — see
  `memory/feedback_no_eslint_plugin_crypto.md`).
- Oxlint stock has a partial node-related ruleset.

**Where we add depth:** explicit security framing (vs general best
practice), CWE-mapped rules, the crypto consolidation that makes this
the single source for Node + crypto rules.

---

### `eslint-plugin-jwt`

**Our scope:** JSON Web Token correctness — algorithm pinning,
verify-before-decode, expiration enforcement, hardcoded secret detection.

**Neighbors in this space:**
- No dedicated JWT lint plugin in the ecosystem we're aware of.
- Snyk Code and Semgrep detect a few JWT issues at the SAST tier.

**Where we add depth:** the whole vertical — open space where Interlace
has community-leadership opportunity. Worth investing in coverage breadth
and visibility.

---

### `eslint-plugin-express-security`

**Our scope:** Express.js application security — middleware ordering,
unsanitized route params, missing helmet, OWASP Top 10 mapped per rule.

**Neighbors in this space:**
- `helmet` (~3M weekly) — runtime middleware, not lint, but solves
  adjacent problems and is commonly bundled.
- `eslint-plugin-security-node` has a few Express-adjacent rules.

**Where we add depth:** lint-time enforcement of patterns helmet enables
at runtime, plus the things helmet doesn't cover (route param handling,
session config, error-leak prevention). Mostly open space.

---

### `eslint-plugin-lambda-security`

**Our scope:** AWS Lambda function security — IAM scoping hints, timeout
ceilings, environment-variable handling, cold-start patterns.

**Neighbors in this space:**
- `cfn-lint`, `checkov`, `tflint` — operate on infrastructure-as-code
  (CloudFormation, Terraform), not the function source.

**Where we add depth:** the only lint-time peer at this resolution for
function-source patterns. Open space.

---

### `eslint-plugin-mongodb-security`

**Our scope:** NoSQL injection prevention, Mongoose patterns, unsafe
query construction.

**Neighbors in this space:**
- No lint-time peer we're aware of.
- Mongoose itself has runtime schema validation.
- Semgrep and CodeQL detect a portion of NoSQL injection at the
  SAST tier.

**Where we add depth:** the only lint-time NoSQL safety story today.
Open space.

---

### `eslint-plugin-nestjs-security`

**Our scope:** NestJS-specific patterns — Controllers, Providers,
Decorators, Guards, security-aware DI patterns.

**Neighbors in this space:**
- `eslint-plugin-nestjs` (small, low traffic) — generic best-practices,
  not security-focused.

**Where we add depth:** the security framing for the NestJS framework.
Open space.

---

### `eslint-plugin-vercel-ai-security`

**Our scope:** Vercel AI SDK usage — prompt injection guards, tool
confirmation, output filtering, hardcoded API keys.

**Neighbors in this space:**
- Emerging neighborhood. No dedicated LLM-security lint plugin yet.
- Generic LLM safety is typically handled at the SDK or via Semgrep
  custom rules.

**Where we add depth:** the first lint-time AI-SDK safety library.
Strong opportunity to define the conventions for this neighborhood
as the ecosystem matures.

---

### `eslint-plugin-pg`

**Our scope:** PostgreSQL interaction safety — parameterized queries,
taint tracking around `pg.query`, common SQL-injection patterns in
JS/TS code.

**Neighbors in this space:**
- No lint-time peer.
- Semgrep and CodeQL cover SQL injection at the SAST tier with taint
  analysis.

**Where we add depth:** lint-tier (PR-time, pre-commit) SQL safety with
Postgres-specific awareness. SAST tools run deeper but slower; we run
faster but shallower — complementary, not overlapping.

## General-purpose plugins (our 8)

The code-quality and developer-experience neighborhood is crowded.
Community has done strong work here for years. Where neighbors cover a
domain well, our value is engine portability + CWE/CVSS metadata +
MCP integration — not replacing them.

### `eslint-plugin-maintainability`

**Our scope:** complexity ceilings (cognitive + cyclomatic), dead code
hints, readability guardrails.

**Neighbors in this space:**
- `eslint-plugin-sonarjs` (~1.9M weekly) — strong on cognitive complexity.
- ESLint core (`complexity`, `max-lines`, `max-depth`, etc.).
- `eslint-plugin-unicorn` (~5.5M) — overlaps in a small subset.
- Oxlint stock has complexity-style rules.

**Where we add depth:** integration with our other plugins (security
rules can reference maintainability budgets), per-rule metadata. This
is a well-served neighborhood — we contribute rather than displace.

---

### `eslint-plugin-reliability`

**Our scope:** defensive programming, error handling, async correctness,
promise hygiene.

**Neighbors in this space:**
- `eslint-plugin-promise` (~5.6M weekly) — dedicated promise patterns,
  community standard.
- `eslint-plugin-fp` — functional safety patterns.
- Subsets of `eslint-plugin-unicorn` cover null/undefined safety.
- Oxlint stock covers many promise patterns.

**Where we add depth:** runtime-stability framing across paradigms (not
just FP), structured error handling. Crowded neighborhood, strong peers.

---

### `eslint-plugin-modernization`

**Our scope:** prefer modern ES idioms (`Array.at`, optional chaining,
nullish coalescing, etc.) over legacy patterns.

**Neighbors in this space:**
- `eslint-plugin-unicorn` (~5.5M) — the dominant modernization plugin.
- ESLint core has built-in deprecation rules.
- Oxlint stock mirrors a meaningful slice of unicorn.

**Where we add depth:** more selective recommendations (unicorn is
deliberately opinionated; we ship a curated subset with metadata).
Heavily crowded — this is a "contribute, don't replace" neighborhood.

---

### `eslint-plugin-conventions`

**Our scope:** project conventions — naming, file structure, code style.

**Neighbors in this space:**
- `eslint-plugin-perfectionist` (~3M+ weekly, growing) — organization
  and ordering.
- `eslint-plugin-prettier` (~23.8M) — formatting (different category).
- ESLint core naming rules.
- Oxlint stock has a conventions namespace.

**Where we add depth:** project-level conventions that span files
(not just within-file), engine portability. Crowded.

---

### `eslint-plugin-modularity`

**Our scope:** module boundaries, circular-dependency detection,
layered-architecture enforcement.

**Neighbors in this space:**
- `eslint-plugin-import` (~38.2M weekly) — includes cycle detection
  (`import/no-cycle`).
- `eslint-plugin-boundaries` — layered-architecture rules.
- Nx boundary rules — monorepo-specific.
- `eslint-plugin-canonical` — overlap on architecture rules.
- `eslint-plugin-perfectionist` (~3M+) — organization overlap.

**Where we add depth:** faster cycle detection (the `import/no-cycle`
slowness on monorepos is a known community pain point — see
benchmarks). Engine portability. Moderately crowded with one giant
neighbor.

---

### `eslint-plugin-operability`

**Our scope:** observability hooks, structured logging, runtime
resilience patterns.

**Neighbors in this space:**
- Almost none in lint space today.
- Organizations typically write custom org-internal rules.

**Where we add depth:** community-leadership opportunity in a mostly
empty neighborhood. Worth growing into a recognized standard.

---

### `eslint-plugin-react-a11y`

**Our scope:** React-specific WCAG / accessibility rules.

**Neighbors in this space:**
- `eslint-plugin-jsx-a11y` (~23.7M weekly) — the dominant
  React-a11y plugin.
- `eslint-plugin-react` (~34M) — a11y subset.
- `axe-core` — runtime, not lint.

**Where we add depth:** integration with our React-features rules,
engine portability, CWE-adjacent disability-impact metadata. The
jsx-a11y plugin is well-loved community standard — we contribute
focused additions, not a replacement.

---

### `eslint-plugin-react-features`

**Our scope:** advanced React patterns — hooks usage, render patterns,
state-management hygiene.

**Neighbors in this space:**
- `eslint-plugin-react-hooks` (~52.9M weekly) — official React team plugin,
  community standard for `exhaustive-deps` and `rules-of-hooks`.
- `eslint-plugin-react` (~34M) — broad React patterns.
- `eslint-plugin-react-refresh` — Fast Refresh compatibility.
- `@next/eslint-plugin-next` (~28.2M) — Next.js specific.

**Where we add depth:** patterns beyond what the React team's plugin
covers (e.g., performance anti-patterns, concurrent-rendering pitfalls).
Heavily crowded with multiple dominant neighbors — careful curation
matters here.

---

### `eslint-plugin-import-next`

**Our scope:** modern import sorting, validation, architectural boundary
enforcement.

**Neighbors in this space:**
- `eslint-plugin-import` (~38.2M weekly) — the long-standing community
  default. The `import/no-cycle` rule is famously slow on large monorepos
  (one of the most-upvoted ESLint plugin issues).
- `eslint-plugin-unused-imports` (~5.6M) — focused unused-import removal.
- `eslint-plugin-simple-import-sort` (~3.9M) — ordering with autofix.
- `eslint-plugin-perfectionist` (~3M+, rising) — organization
  including imports.
- `Knip` / `unimported` — CLI tools, different tier.

**Where we add depth:** much faster cycle detection, modern flat-config
native, designed for Oxlint host runtime. Heavily crowded but with a
clear performance contribution to make.

## Cross-cutting neighbor: `typescript-eslint`

`typescript-eslint` (~30M+ weekly across the `@typescript-eslint/eslint-plugin`
and `@typescript-eslint/parser` packages combined) is a special case. It's both:

- The **parser stack** every TypeScript-aware ESLint plugin (including
  ours) depends on, and
- A rule library in its own right, with ~150 rules covering type-aware
  safety (`no-unsafe-*`, `no-misused-promises`, `prefer-readonly`,
  `strict-boolean-expressions`, etc.).

**Where they shine:** the deep type-aware tier. The `no-unsafe-*` family
catches whole classes of type-escape bugs that no AST-only rule can
detect. The maintainers shape the TypeScript-ESLint integration the
entire community depends on.

**Where Interlace adds depth:** domain framing on top of the type
layer. `pg/no-unsafe-query` and `mongodb-security/no-unsafe-query` use
type-awareness for taint tracking but lift the analysis into a
database-safety contract; `typescript-eslint/no-unsafe-assignment`
stops at the type level. Same substrate (type graph), different
abstraction. We are complementary, not overlapping.

**Engine portability note:** typescript-eslint is ESLint-only by
design (it ships its own parser). Our deep-tier rules share the same
constraint today; per `INTEROP_PHILOSOPHY.md`, the TSC native plugin
host is the long-horizon second runtime for both libraries.

## Adjacent tooling (not ESLint plugins, but cover overlapping needs)

Several non-plugin tools cover use cases users sometimes consider as
alternatives to plugins in our space. Worth knowing.

### Unused-import / dead-code tools

- **`knip`** (~700k weekly, rising) — unused files, exports, and
  dependencies at the project graph level. Different tier from
  `import-next/no-unused-imports` (file-local). Complementary: knip
  for periodic full-project audits, import-next for per-edit
  lint-time.
- **`unimported`** — similar use case, less active.
- **`dependency-cruiser`** — module dependency graph linting,
  architecture rules. Overlaps with `modularity` on layered-architecture
  enforcement, with a different invocation model (CLI tool vs ESLint
  plugin). Both are valid; users typically pick one.

### Linting aggregators

- **`trunk check`** (Trunk.io) — bundles many linters (ESLint, Oxlint,
  Semgrep, security tools) behind a single workflow. Not a competitor
  to plugins — they ship and re-distribute plugins like ours. Worth
  visibility for them to surface Interlace in their default config.
- **`megalinter`**, **`super-linter`** (GitHub) — similar aggregator
  category. Same logic: they consume plugins.

### Runtime security middleware (mentioned for context)

- **`helmet`** (~3M weekly) — Express runtime security headers. Covers
  some patterns our `express-security` lints at compile time, but at
  request-time. Complementary; users typically run both.
- **`mongoose`** runtime schema validation — covers a chunk of what
  `mongodb-security` lints. Different tier (runtime vs lint-time).

### Standalone CLI security analyzers

- **`gitleaks`**, **`trufflehog`** — secret scanning in git history /
  filesystem. Overlap with `secure-coding/no-hardcoded-credentials`
  but operate on a different surface (history scan vs source-tree
  lint). Most repos run both.

These tools are not on the "compete or collaborate" axis — they sit at
a different tier of the pipeline. Worth mentioning so the landscape map
isn't surprising to a reader who reaches for them.

## Open positioning summary

Three places where Interlace is well-positioned to lead the community
conversation:

1. **Domain-security depth.** Nine dedicated security plugins covering
   verticals (pg, mongodb, jwt, express, lambda, nestjs, vercel-ai)
   where no peer ships a comparable depth today. Open neighborhoods.
2. **Engine portability contract.** The only rule library with
   CI-enforced diagnostic parity across two engines (ESLint + Oxlint)
   today, with Biome and TSC native (Go) on the roadmap. See
   `INTEROP_PHILOSOPHY.md`.
3. **AI / agent integration.** Per-rule CWE / CVSS / OWASP metadata,
   11 MCP servers, SARIF output. No peer in the lint ecosystem ships
   this systematically.

In the general-purpose categories (maintainability, reliability,
modernization, conventions, react-features, react-a11y, imports), the
community already has strong neighbors. Our contribution there is
the engine-portability story + metadata, not replacement.

  the npm registry pulls.

  crosses 100k weekly downloads, or when a peer that was abandoned
  resumes activity.
